基本操作三部曲

git init---仓库初始化

git add---刷新head头文件文件

git commit---完成提交

git init

创建版本库，初始化仓库操作，如果clone仓库则无须如此，因为文件/文件夹已经存在版本库信息了，即存在.git文件夹版本库文件

初始化版本库时，git 1.6.5或最新版本可以直接输入目录完成初始化即：git init testDir

git add

当我们 修改了一个文件并add，其实只是修改了 head文件头，而没有加入到 暂存区(stage)中, ps：此时 git diff 不带参数比的是 工作区与暂存区差异，所以是空的， 此时若 调用 git status 则会提示： 用 reset 撤销提交，否则就准备 提交到 暂存区啦；来真的咯；

git commit

当我们 修改了一个文件并add，其实只是修改了 head文件头，而没有加入到 暂存区(stage)中, ps：此时 git diff 不带参数比的是 工作区与暂存区差异，所以是空的， 此时若 调用 git status 则会提示： 用 reset 撤销提交，否则就准备 提交到 暂存区啦；来真的咯；

暂存区

原理、概念、替换本地改动等

首先 必须明白 3个区： 工作区 --- 暂存区 --- HEAD头区，一个 文件实际上 存在 3种版本（工作区、暂存区、版本库（当前分支）中最新版本）,举例说明：

* 文件内容 存在 git对象库里，.git/objects目录中，文件索引 建立了 文件 和 对象库中 对象和实体的 映射对应, 见下图：

* commit提交, 暂存区的 目录树 会写到 版本库 对象中， HEAD会做相应的更新， 即 HEAD最新指向的目录树 就是提交时 原暂存区的目录树.

比较差异Diff

.

git diff HEAD 则会出现修改差异（比的是工作区 和 当前工作分支头）

* 当调用 git diff --cached 或者 --staged 则是查看 暂存区 和 版本库之间差异；如何查看提交成功， 只要有日志 提交id 和 说明，即为提交成功；

重置Reset

* 当执行 reset HEAD，暂存区的 目录会被重写，即被 HEAD 目录指向 替换掉，工作区不受影响，换而言之 暂存区 和 HEAD就是 2个缓冲对象，一个 对工作区，一个对分支头，而两者之间又有 关系交互；

checkout暂存区操作

* 当执行 git checkout . 或者 git checkout -- 会用暂存区 全部文件或者指定文件 替换工作区的 文件，危险性就是 会清除 工作区中未 添加到 暂存区的 改变；

* 同理可得，当执行 git checkout HEAD . 或者 git checkout HEAD 会用HEAD 分支 指向的文件 替换 暂存区 和 工作区 指定或者全部文件；会清除 工作区未提交的改动，也会清除暂存区未提交的改动；
